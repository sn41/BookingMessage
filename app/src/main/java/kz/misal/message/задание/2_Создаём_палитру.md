В вашем примере показана типичная ситуация, когда разработчик пытается перенести названия слоев из Figma напрямую в код. Это работает на старте, но быстро превращается в хаос.

**Критика вашего примера**
- Смешивание абстракций: У вас в одном файле лежат «буквальные» цвета (ForestGreen, Sapphire) и цвета с привязкой к логике (LightGrayAlto, где Alto — это оттенок). А в самой схеме появляются уже третьи названия (BrandBlueAction, StarYellow), которых даже нет в списке переменных.

- Проблема темной темы: Как назвать цвет ForestGreen, если в темной теме зеленый статус должен быть светлее, чтобы читаться на черном фоне? ForestGreenLight? А если их три оттенка? - Привязка к объектам: Название StarYellow (Желтая звезда) — это антипаттерн. Если завтра дизайнер решит использовать этот же желтый цвет для плашки «Скидка», название StarYellow потеряет смысл."

## Двухуровневая система (Примитивы + Семантика)

- Уровень 1. Файл Color.kt (Примитивы / Тоновая шкала)
Здесь мы описываем «банки с краской». Никакой логики использования.
В M3 золотой стандарт — называть цвета по Базовому оттенку + Уровню яркости (от 0 до 100), где 0 — черный, 100 — белый.

- Уровень 2. Файл Theme.kt (Семантика / Схема)
  Здесь мы берем наши «банки с краской» и распределяем их по ролям (слотам). Схема ничего не знает о том, что кнопка синяя, она знает, что кнопка — это primary.

```kotlin

//package kz.misal.message.ui.theme
//import androidx.compose.ui.graphics.Color

// Уровень 1

// Базовые (Нейтральные)
val Neutral10 = Color(0xFF1A1A1A) // Бывший Charcoal
val Neutral20 = Color(0xFF222222) 
val Neutral40 = Color(0xFF424242) // Бывший DarkGray
val Neutral90 = Color(0xFFE9E9E9)
val Neutral95 = Color(0xFFF5F5F5) // Бывший LightGray

val Blue40 = Color(0xFF003C94) // Ваш Brand
val Blue50 = Color(0xFF006CE4) // Ваш Sapphire
val Blue80 = Color(0xFF6F9BFD) // Ваш CornflowerBlue

// Акцентные и статусные

//import androidx.compose.material3.lightColorScheme

private val LightColorScheme = lightColorScheme(
    // --- ОСНОВНЫЕ ЦВЕТА БРЕНДА ---
    primary = Blue40,               // Главные кнопки (Brand Blue)
    onPrimary = Color.White,        // Текст на главных кнопках
    primaryContainer = Blue10,      // Темно-синяя верхняя панель
    onPrimaryContainer = Color.White,

    // --- ВТОРОСТЕПЕННЫЕ АКЦЕНТЫ ---
    secondary = Yellow50,           // Жёлтый акцент (звезды, бейджи)
    onSecondary = Neutral10,        // Текст на желтом (лучше темный для контраста)
    secondaryContainer = Yellow50.copy(alpha = 0.2f), // Светло-желтый фон
    onSecondaryContainer = Neutral10,

    // --- СТАТУСЫ ---
    tertiary = Green40,             // Зеленый для позитивных статусов (Deal)
    onTertiary = Color.White,
    error = Red40,                  // Красный (Alerts, зачеркнутая цена)
    onError = Color.White,

    // --- ПОВЕРХНОСТИ И ФОНЫ ---
    background = Neutral95,         // Общий фон приложения
    onBackground = Neutral10,       // Основной текст на фоне
    
    surface = Color.White,          // Фон карточек (светлее фона приложения)
    onSurface = Neutral10,          // Текст на карточках
    
    surfaceVariant = Neutral90,     // Серые обводки, фоны неактивных элементов
    onSurfaceVariant = Neutral40    // Второстепенный текст (адрес, приписки)
)
```

# Принципы построения палитры

## 1. Базовые принципы (Фундамент палитры)

Прежде чем думать о слотах M3, вам нужно определить эмоциональное и функциональное ядро проекта. Возьмите за основу классическое правило **60-30-10** и определите «трех китов»:

1.  **Фон и Поверхности (60%):** Это ваш холст.
    * *Светлая тема:* Выберите чистый белый (`#FFFFFF`) или слегка тонированный серый/кремовый (как `#E3E3E3` в нашем примере).
    * *Тёмная тема:* Избегайте чисто черного (`#000000`). Используйте глубокие темно-серые или темно-синие оттенки (например, `#121212` или `#0F172A`). Это снижает нагрузку на глаза и позволяет теням (elevation) работать корректно.
2.  **Primary — Цвет Бренда (30%):** Главный акцентный цвет.
    * Определяет индивидуальность приложения. Используется для главных кнопок (Call to Action), активных состояний вкладок, важных иконок.
    * *Совет:* Всегда проверяйте его на контрастность с фоном. Если он слишком светлый, вам понадобится более темный оттенок для текста на нем (`onPrimary`).
3.  **Secondary — Вторичный акцент (10%):**
    * Цвет, который поддерживает `Primary`, но не спорит с ним. Часто это комплементарный цвет (напротив на цветовом круге) или аналоговый (соседний).
    * Используется для бейджей, рейтингов (например, желтые звезды), плавающих кнопок (FAB), переключателей.

> **Pro-совет:** Для генерации тональных палитр (Primary, Secondary, Tertiary и их контейнеров) из этих базовых цветов используйте плагин **Material Theme Builder** в Figma. Он автоматически рассчитает нужные уровни яркости для светлой и темной тем, избавив вас от рутинной работы.

## 2. Что делать с цветами вне логики M3?

В стандартной схеме M3 есть `Primary`, `Secondary`, `Tertiary`, `Error`, `Background` и `Surface`. Но в реальном коммерческом приложении вам часто нужны:
* **Цвета статусов:** Успех (Зеленый), Предупреждение (Желтый/Оранжевый), Информация (Синий/Голубой). M3 предлагает "из коробки" только `Error` (Красный).
* **Брендовые цвета партнеров:** Синий для VK, красный для YouTube, зеленый для WhatsApp. Они не должны подстраиваться под ваш бренд.
* **Специфические маркетинговые теги:** Уникальные цвета (например, золотой или фиолетовый для подписки «Premium»), которые нельзя привязать к базовой логике темы.

**Главное правило:** Никогда не используйте "хардкод" (прямое указание `Color(0xFF...)` прямо в верстке экрана), даже если цвет кажется уникальным. Это сломает поддержку темной темы и превратит рефакторинг в кошмар.

Вместо этого мы **расширяем** стандартную тему M3 с помощью паттерна `CompositionLocal`.

### Архитектура кастомных цветов в Compose

Внедрение дополнительных цветов состоит из четырех шагов. Лучше всего разместить этот код в файле `Theme.kt` или вынести в отдельный файл `CustomColors.kt`.

#### Шаг 1: Создаем модель данных и провайдер
Определяем логические группы для наших нестандартных цветов. Обратите внимание: для каждого семантического цвета (например, `success`) мы обязательно создаем пару для контента на нем (`onSuccess`), чтобы текст всегда оставался читаемым.

```kotlin
//import androidx.compose.runtime.staticCompositionLocalOf
//import androidx.compose.ui.graphics.Color

// 1. Описываем структуру наших дополнительных цветов
data class CustomColors(
    val success: Color,
    val onSuccess: Color,
    val warning: Color,
    val onWarning: Color,
    val premiumBadge: Color,
    val socialVk: Color // Цвета брендов часто статичны
)

// 2. Создаем глобальный ключ (провайдер) с заглушками
val LocalCustomColors = staticCompositionLocalOf {
    CustomColors(
        success = Color.Unspecified,
        onSuccess = Color.Unspecified,
        warning = Color.Unspecified,
        onWarning = Color.Unspecified,
        premiumBadge = Color.Unspecified,
        socialVk = Color.Unspecified
    )
}
```

#### Шаг 2: Создаем удобный доступ (Extension)
Чтобы обращаться к новым цветам так же легко, как к стандартным (`MaterialTheme.colorScheme...`), добавляем свойство-расширение:

```kotlin
//import androidx.compose.material3.MaterialTheme
//import androidx.compose.runtime.Composable
//import androidx.compose.runtime.ReadOnlyComposable

val MaterialTheme.customColors: CustomColors
    @Composable
    @ReadOnlyComposable
    get() = LocalCustomColors.current
```

#### Шаг 3: Задаем палитры для светлой и темной тем
Заполняем наши токены конкретными значениями из тоновой шкалы (файла `Color.kt`).

```kotlin
val lightCustomColors = CustomColors(
    success = Green40,       // Насыщенный зеленый для светлого фона
    onSuccess = Color.White,
    warning = Yellow50,
    onWarning = Neutral10,   // Темный текст для контраста с желтым
    premiumBadge = Purple50,
    socialVk = Color(0xFF0077FF) // Статичный бренд-цвет
)

val darkCustomColors = CustomColors(
    success = Green80,       // Пастельный зеленый для темного фона
    onSuccess = Neutral10,   // Темный текст на пастельном фоне
    warning = Yellow80,
    onWarning = Neutral10,
    premiumBadge = Purple80,
    socialVk = Color(0xFF0077FF) // Оставляем неизменным
)
```

#### Шаг 4: Интегрируем в `AppTheme`
Оборачиваем стандартную тему в наш кастомный провайдер.

```kotlin
@Composable
fun AppTheme(
    darkTheme: Boolean = isSystemInDarkTheme(),
    content: @Composable () -> Unit
) {
    val colorScheme = if (darkTheme) DarkColorScheme else LightColorScheme
    val customColors = if (darkTheme) darkCustomColors else lightCustomColors

    // Передаем наши кастомные цвета вниз по дереву
    CompositionLocalProvider(
        // Перечисляем через запятую
        LocalCustomColors provides customColors
    ) {
        // Вызываем стандартную тему M3 внутри
        MaterialTheme(
            colorScheme = colorScheme,
            typography = AppTypography,
            shapes = AppShapes,
            content = content
        )
    }
}
```

### Как это выглядит в верстке экрана

Теперь использование нестандартных цветов в любом компоненте выглядит профессионально, безопасно и автоматически реагирует на смену темы:

```kotlin
Text(
    text = "Оплата прошла успешно",
    color = MaterialTheme.customColors.onSuccess, // Белый днем, темный ночью
    modifier = Modifier
        .background(MaterialTheme.customColors.success) // Насыщенный днем, пастельный ночью
        .padding(16.dp)
)
```

### Резюме подхода:
1.  **Максимально используйте базу:** Если цвет может быть `surfaceVariant` (для неактивных подложек) или `tertiary` (для дополнительного акцента) — используйте их. Не плодите новые сущности без крайней необходимости.
2.  **Изолируйте семантику:** Статусы (`Success`, `Warning`, `Info`) — первые и главные кандидаты на вынесение в `CustomColors`.
3.  **Уважайте чужие бренды:** Если цвет (например, логотип партнера VK или Google) должен быть строго определенного HEX-кода и днем, и ночью, создайте для него токен в `CustomColors` и просто передайте одно и то же значение в обе сборки (light и dark).
