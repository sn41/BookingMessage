# Откуда берутся элементы, которые необходимо показывать?

Когда мы переходим от создания одного экрана к разработке реального коммерческого продукта, управление контентом становится важнее самого кода.

В рамках грамотного управления жизненным циклом приложения (Application Lifecycle Management) 
такие элементы интерфейса, как Message, никогда не зашиваются жестко в мобильный клиент.

Если при каждой смене запятой или иконки вам придется выпускать новое обновление в Google Play, разработка остановится.

Вот как этот процесс организован в крупных IT-компаниях (например, в Booking, чей макет мы разбираем).

### 1. Архитектура

Профессиональный подход заключается в **строгом разделении UI-компонента и данных**.

Мы уже сделали первый шаг: создали универсальный компонент `Message` на основе Slot API. 
Он ничего не знает о том, *какой* текст внутри и *какая* там иконка. 
Он просто умеет красиво отрисовывать то, что ему дадут.

Второй шаг — это **Server-Driven UI (SDUI)** или передача размеченного текста с бэкенда. 
Мобильное приложение не должно хранить тексты про комиссии в `strings.xml`. 
Бэкенд присылает готовый JSON, который описывает, как собрать сообщение:

```json
{
  "message_type": "info_warning",
  "icon": "ic_union",
  "content": "Commission paid on bookings...",
  "links": [
    {
      "text": "Find out more",
      "url": "https://www.booking.com/ranking_info",
      "style": "underlined" // или "plain"
    }
  ]
}
```
Мобильный разработчик пишет парсер, который один раз читает этот JSON и автоматически собирает те самые `AnnotatedString` с нужными стилями (`withStyle(SpanStyle(textDecoration = ...))`) и подставляет нужную иконку в слот `leadingIcon`.

*Альтернативный вариант:* Бэкенд присылает просто HTML-строку (`"Текст... <a href='...'>Find out more</a>"`), а в Compose пишется утилита, которая конвертирует HTML-теги в `AnnotatedString`.

### 2. Кто внедряет эти элементы (Роли в команде)

В продуктовой разработке создание такого сообщения проходит через четкий конвейер:

* **Юристы / Compliance:** Это они инициируют появление таких плашек. Текст про комиссию и ранжирование (ranking parameters) — это юридическое требование Евросоюза (DSA) или других регуляторов. Они диктуют точные формулировки вплоть до буквы.
* **Product Manager (PM) / Копирайтер:** Переводят юридический язык в более-менее читаемый пользовательский интерфейс. Они же решают, на каких экранах это сообщение должно появиться.
* **UX/UI Дизайнер:** Проектирует внешний вид в дизайн-системе. Дизайнер определяет правила: *"Если это юридический текст, ссылка должна быть с подчеркиванием, чтобы её точно заметили. Если это маркетинговая ссылка, подчеркивание убираем, чтобы не перегружать дизайн"*. Он же рисует иконки (info, warning, error) и передает их разработчикам.
* **Backend-разработчик:** Заводит этот текст в админку (CMS) или базу данных, чтобы API могло отдавать его мобильному приложению в зависимости от страны пользователя.
* **Mobile-разработчик (Вы):** Создает надежный UI-компонент, который может переварить любые данные с бэкенда, не ломая верстку, и корректно открывает ссылки.

### 3. Как часто они могут меняться?

**Очень часто.** Это самая динамичная часть приложения.

1.  **A/B тестирование:** Маркетологи постоянно проверяют гипотезы. Сегодня ссылка называется *"Find out more"* и она синяя без подчеркивания. 
Завтра они запускают тест на 50% пользователей, где ссылка называется *"Read full policy"*, она жирная и с подчеркиванием, чтобы проверить, станут ли люди кликать чаще.
2.  **Локализация:** Текст переводится на десятки языков. Длина немецкого текста может быть в два раза больше английского. 
Ваш UI должен растягиваться автоматически (поэтому мы использовали `weight(1f)` и выравнивание по первой строке).
3.  **Законодательство:** Требования регуляторов меняются каждый год. 
Если текст зашит в приложении (`strings.xml`), пользователи со старыми версиями приложения не увидят обновленный текст, и компания получит многомиллионный штраф. 
Бэкенд-подход позволяет обновить текст мгновенно у всех пользователей.

**Итог:** Ваша задача как Android-инженера — создать такую дизайн-систему (универсальный `Message` и `Typography`), 
чтобы при любых изменениях текста, иконок или стилей ссылок со стороны бэкенда или дизайнеров, 
вам вообще **не приходилось трогать код Compose-компонента**.
