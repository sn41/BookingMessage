Однако базовый этап создания UI — это только верхушка айсберга. Если мы хотим, чтобы ваши студенты были готовы к реальному production (или к победам на WorldSkills), есть несколько критически важных методологических столпов, которые пока остались за кадром.

Вот что я бы обязательно добавил в следующие модули курса:

### 1. Архитектурный паттерн UDF (Unidirectional Data Flow)
Мы научили студентов рисовать красивые карточки отелей и собирать их в списки. Но мы пока не говорили о том, **откуда** эти карточки берут данные и **как** они обновляются.

* **Проблема новичков:** Студенты часто пытаются менять данные прямо внутри UI (например, передают объект `Hotel`, и по клику на кнопку пишут `hotel.isBooked = true`). Это путь в никуда.
* **Методология:** Нужно внедрить концепцию **Однонаправленного потока данных (UDF)**.

  Студенты должны вызубрить мантру: *«UI — это глупое отражение State. UI никогда не меняет данные сам»*. Компонент (`Screen`) только генерирует события (Intent/Action), отправляет их во `ViewModel`, а `ViewModel` возвращает новый иммутабельный `State`, который заставляет UI перерисоваться.

### 2. Дизайн «Несчастливого пути» (The Unhappy Path)
В наших примерах (Figma, JSON-ответы) всегда всё идеально: картинки загружены, тексты помещаются в рамки, интернет работает со скоростью света.

* **Методология:** Настоящий инженер проектирует экраны по паттерну **LCE (Loading, Content, Error)**.
* Что нужно показать студентам:
    * **Loading:** Как делать красивые скелетные загрузки (Shimmer эффекты) вместо раздражающих крутящихся спиннеров (ProgressBar).
    * **Error:** Как элегантно обрабатывать отсутствие сети. Что показать, если SDUI-сервер прислал битый JSON?
    * **Empty State:** Как выглядит экран "Избранное", если пользователь еще ничего туда не добавил? (Пустые состояния — это важный инструмент удержания пользователя, там должна быть кнопка "Перейти к поиску").

### 3. Культура Доступности (Accessibility / A11y)
Мы вскользь упомянули M3 тональные шкалы, которые гарантируют контрастность. Но доступность в Android — это огромный и обязательный пласт работы в западных и крупных отечественных компаниях.

* **Чему нужно научить:**
    * **Правило 48.dp:** Любая кликабельная область (даже если сама иконка 24.dp) должна иметь размер касания минимум 48x48 dp. Иначе пользователи с крупными пальцами или в транспорте будут промахиваться. В Compose это делается через `Modifier.minimumInteractiveComponentSize()`.
    * **Screen Readers (TalkBack):** Студенты должны понимать, зачем мы везде писали `contentDescription = null` для декоративных иконок, и почему для значимых иконок туда нужно писать локализованный текст. Как приложение будет "звучать" для слепого пользователя?

### 4. Анатомия Рекомпозиции (Производительность Compose)
Мы затронули `derivedStateOf`, чтобы избежать лишних перерисовок. Но это лишь следствие. Студентам нужна прочная ментальная модель того, как Compose обновляет экран.

* **Опасность:** Compose может перерисовывать функции сотни раз в секунду. Если студент напишет сортировку списка или тяжелые вычисления прямо внутри `@Composable` функции, приложение будет жутко тормозить при скролле.
* **Концепции для изучения:**
    * Что такое `Stable` и `Immutable` структуры данных.
    * Почему передавать `List<String>` в компонент — это потенциальная проблема (Compose считает стандартный `List` нестабильным и всегда перерисовывает его), и как использовать `kotlinx.collections.immutable`.
    * Использование Android Studio Layout Inspector для поиска лишних рекомпозиций.

### 5. Локализация как архитектурный принцип
В разделе про SDUI мы немного обсудили, что тексты не должны быть зашиты в код. Но даже статичные тексты интерфейса (заголовки табов, кнопки "Отмена", "ОК") требуют правильного подхода.

* **Методология:** Строгий запрет на хардкод строк в верстке (`Text("Привет")`).
* **Инструменты:** Использование `strings.xml`, передача `stringResource(R.string...)`. Работа с плюрализацией (когда нужно правильно написать "1 отзыв", "2 отзыва", "5 отзывов" — в Android для этого есть механизм Plurals, который многие студенты игнорируют, пытаясь писать страшные `if/else`).

# Pluralis

**Плюрализация (Plurals)** — это один из самых важных и одновременно недооцененных новичками механизмов в Android.

Проблема, которую он решает, знакома каждому. Представьте, что вы выводите количество отзывов у отеля.
* В английском всё просто: 1 review, 2 review**s**, 5 review**s** (всего две формы: единственная и множественная).
* А теперь посмотрите на русский язык: 1 отзыв, 2 отзыв**а**, 5 отзыв**ов**, 21 отзыв, 114 отзыв**ов**.

### Как делают новички (Антипаттерн)

Студенты обычно пытаются решить это «в лоб» через `if/else` или `when` прямо в коде:

```kotlin
// ПЛОХО! Никогда так не делайте в production.
val text = when {
    count % 10 == 1 && count % 100 != 11 -> "$count отзыв"
    count % 10 in 2..4 && count % 100 !in 12..14 -> "$count отзыва"
    else -> "$count отзывов"
}
```
**Почему это ужасно?** Потому что 
- в арабском языке 6 форм множественного числа, 
- в польском — свои правила, 
- а в японском множественного числа для объектов вообще нет. 

Если вы захардкодите эту математику в Kotlin, ваше приложение невозможно будет перевести на другие языки.

---

### Как делают профессионалы (Android Plurals)

В Android есть встроенный математический и лингвистический движок, который сам знает правила всех языков мира. Ваша задача — просто дать ему варианты слов.

#### Шаг 1: Описываем формы в XML

В файле `res/values/strings.xml` (и в его переводах, например, `res/values-ru/strings.xml`) мы используем специальный тег `<plurals>`.

**Для английского (`res/values/strings.xml`):**
```xml
<plurals name="reviews_count">
    <item quantity="one">%d review</item>
    <item quantity="other">%d reviews</item>
</plurals>
```

**Для русского (`res/values-ru/strings.xml`):**
```xml
<plurals name="reviews_count">
    <item quantity="one">%d отзыв</item>
    
    <item quantity="few">%d отзыва</item>
    
    <item quantity="many">%d отзывов</item>
    
    <item quantity="other">%d отзыва</item>
</plurals>
```
*Примечание: `%d` — это плейсхолдер (заглушка), куда система подставит само число.*

#### Шаг 2: Вызов в Jetpack Compose

В Compose есть специальная функция `pluralStringResource`. Она делает всю магию под капотом.

```kotlin
//import androidx.compose.ui.res.pluralStringResource

@Composable
fun ReviewBadge(count: Int) {
    // Вызываем плюрализацию
    val reviewText = pluralStringResource(
        id = R.plurals.reviews_count,
        count = count, // 1. Это число говорит Android, какое правило языка выбрать (one/few/many)
        count          // 2. А это число подставится вместо %d в самом тексте
    )

    Text(text = reviewText)
}
```

**Частый вопрос от студентов:** *"Почему мы передаем переменную `count` два раза подряд?"*
**Ответ:** 
- Первый `count` — это параметр для логического движка Android (чтобы он понял, что 5 — это `many`). 
- Второй `count` — это фактическое значение, которое напечатается на экране вместо `%d`. Вы могли бы написать так: `pluralStringResource(id, count, "Ого, целых $count")`, если бы у вас в XML стояло `%s отзывов`.

### Как Plurals сочетается с Server-Driven UI (SDUI)?

Поскольку мы много говорили об архитектуре данных, здесь возникает интересный конфликт.

Если вы используете **SDUI**, мобильное приложение **вообще не должно** использовать `R.plurals`.
Почему? Потому что философия SDUI гласит: мобильный клиент — глупый.

Если бэкенд присылает данные так:
```json
{
  "reviews_count": 142
}
```
Тогда мобильному приложению придется самому использовать `pluralStringResource`. Это нарушает SDUI (клиент знает о бизнес-логике отображения).

**Правильный SDUI-бэкенд** сам прогоняет число через библиотеку локализации (например, *ICU MessageFormat* на сервере) и отдает на телефон уже готовую строку для конкретного языка:
```json
{
  "reviews_text": "142 отзыва" 
}
```
Клиенту остается только засунуть эту строку в `Text(text = component.reviewsText)`.

### Резюме для студентов

1.  Никогда не хардкодьте окончания слов через `if/else`.
2.  Используйте `<plurals>` в ресурсах для статических текстов (когда приложение само считает данные).
3.  Обязательно проверяйте файл перевода (в английском достаточно `one` и `other`, а в русском обязательны `one`, `few`, `many` и `other`).
4.  Если переходите на строгий SDUI — забирайте логику склонений на сервер!
