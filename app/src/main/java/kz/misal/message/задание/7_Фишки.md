
### Фишка 1: Магия простых анимаций (`animate*AsState`)

В старом Android (XML) создание анимации цвета или размера требовало написания громоздких XML-файлов или сложных `ValueAnimator`. 
В Compose анимация базовых свойств делается добавлением **одного слова**.

Например, вы хотите, чтобы кнопка закладки (Save) плавно меняла цвет с серого на красный при нажатии, а иконка слегка увеличивалась.

```kotlin
//import androidx.compose.animation.animateColorAsState
//import androidx.compose.animation.core.animateDpAsState
//import androidx.compose.animation.core.tween
//import androidx.compose.foundation.clickable
//import androidx.compose.foundation.layout.size
//import androidx.compose.material.icons.Icons
//import androidx.compose.material.icons.filled.Favorite
//import androidx.compose.material3.Icon
//import androidx.compose.material3.MaterialTheme
//import androidx.compose.runtime.Composable
//import androidx.compose.runtime.getValue
//import androidx.compose.runtime.mutableStateOf
//import androidx.compose.runtime.remember
//import androidx.compose.runtime.setValue
//import androidx.compose.ui.Modifier
//import androidx.compose.ui.graphics.Color
//import androidx.compose.ui.unit.dp

@Composable
fun AnimatedFavoriteButton() {
    // Два состояния цвета
    var isFavorite by remember { mutableStateOf(false) }

    // 1. Анимируем цвет. Compose сам высчитает все промежуточные оттенки!
    val iconColor by animateColorAsState(
        // Устанавливаем целевой цвет в соответствие двоичному состоянию (а можно и не двоичному!)
        targetValue = if (isFavorite) MaterialTheme.colorScheme.error else MaterialTheme.colorScheme.onSurfaceVariant,
        // задаём кривую перехода
        animationSpec = tween(durationMillis = 300),
        // Метка для отладки
        label = "colorAnimation"
    )

    // 2. Анимируем размер
    val iconSize by animateDpAsState(
        // Устанавливаем размер в соответствие двоичному состоянию
        targetValue = if (isFavorite) 32.dp else 24.dp,
        // задаём кривую перехода, (бывает еще и пружина Ж)
        animationSpec = tween(durationMillis = 300),
        // Метка для отладки
        label = "sizeAnimation"
    )

    // 3. Объект издевательств
    Icon(
        imageVector = Icons.Default.Favorite,
        contentDescription = null,
        tint = iconColor, // Используем анимированное значение
        modifier = Modifier
            .size(iconSize) // Используем анимированное значение
            .clickable { isFavorite = !isFavorite }
    )
}
```
**Почему это круто:** При каждом клике Compose не перерисовывает весь экран, он лишь пересчитывает два конкретных свойства и плавно отрисовывает кадры.

### Фишка 2: Умная производительность (`derivedStateOf`)

Представьте, что у вас есть список отелей (`LazyColumn`). Вы хотите показать кнопку "Наверх" (FAB), когда пользователь проскроллит больше 5 элементов.

Частая ошибка новичков — читать состояние скролла напрямую. 
Это заставляет Compose перерисовывать кнопку "Наверх" при **каждом сдвиге на 1 пиксель**, что убивает батарею и вызывает фризы. 
Решение — `derivedStateOf`.

```kotlin
val listState = rememberLazyListState()

// derivedStateOf кэширует результат. 
// Он заставит UI перерисоваться ТОЛЬКО тогда, когда false сменится на true (на 5-м элементе),
// игнорируя все микро-изменения пикселей при скролле.
val showButton by remember {
    derivedStateOf { listState.firstVisibleItemIndex > 5 }
}

LazyColumn(state = listState) { /* ... */ }

// Плавно показываем или скрываем кнопку
AnimatedVisibility(visible = showButton) {
    FloatingActionButton(onClick = { /* ... */ }) {
        Icon(Icons.Default.KeyboardArrowUp, contentDescription = "Наверх")
    }
}
```

### Фишка 3: CompositionLocal не только для тем

Мы использовали `CompositionLocal` для передачи цветов и отступов сквозь всё дерево компонентов. 
Но профессионалы используют его для внедрения **любых глобальных зависимостей**, 
чтобы не прокидывать их через параметры каждой функции (избегая проблемы "Prop Drilling").

Например, для системы аналитики:
```kotlin
// 1. Создаем провайдер
val LocalAnalyticsTracker = staticCompositionLocalOf<AnalyticsTracker> { 
    error("Analytics не подключена") 
}

// 2. Где-то высоко в иерархии (в MainActivity) передаем реальный объект
CompositionLocalProvider(LocalAnalyticsTracker provides myFirebaseTracker) {
    MyApp()
}

// 3. В любой кнопке на 10-м уровне вложенности просто вызываем:
val analytics = LocalAnalyticsTracker.current
Button(onClick = { analytics.track("Booking_Clicked") }) { ... }
```

Использование `CompositionLocal` для чего-то помимо тем и отступов — это признак взрослой архитектуры. 
Главная проблема, которую решает этот механизм, 
называется **Prop Drilling** (просверливание свойств) — когда вам нужно передать какой-то объект 
через 10 слоев UI-компонентов вниз, хотя промежуточным слоям этот объект совершенно не нужен.

Помимо аналитики, в реальных коммерческих проектах `CompositionLocal` чаще всего используют для следующих 4 задач.

### 1. Глобальный UI-контроллер (Снекбары и Уведомления)

Представьте, что вы находитесь глубоко на экране настроек и хотите показать сообщение об ошибке (Snackbar) в самом низу экрана. В стандартном подходе вам пришлось бы прокидывать коллбэк `onShowError` через все промежуточные функции наверх, к `Scaffold`. Это ужасно засоряет код.

Вместо этого мы создаем локальный провайдер для показа сообщений:

```kotlin
// 1. Создаем провайдер
val LocalSnackbarController = staticCompositionLocalOf<(String) -> Unit> {
    error("Снекбар контроллер не подключен")
}

// 2. В MainActivity или главном экране настраиваем его
@Composable
fun MainScreen() {
    val snackbarHostState = remember { SnackbarHostState() }
    val coroutineScope = rememberCoroutineScope()

    CompositionLocalProvider(
        LocalSnackbarController provides { message ->
            coroutineScope.launch { snackbarHostState.showSnackbar(message) }
        }
    ) {
        Scaffold(snackbarHost = { SnackbarHost(snackbarHostState) }) {
            // ... ваши экраны
        }
    }
}

// 3. Вызываем из любой точки приложения (хоть из маленькой кнопки на 10-м уровне)
@Composable
fun DeleteButton() {
    val showSnackbar = LocalSnackbarController.current
    
    Button(onClick = { 
        // Логика удаления...
        showSnackbar("Отель успешно удален из избранного") 
    }) {
        Text("Удалить")
    }
}
```

### 2. Навигация (Локальный Router)

Прокидывать `NavController` в каждую Composable-функцию — это боль. 
Сигнатуры функций раздуваются, а писать Preview становится невозможно (приходится везде создавать моковые контроллеры).

Прагматичное решение в крупных проектах — вынести роутер в `CompositionLocal`.

```kotlin
val LocalNavController = staticCompositionLocalOf<NavHostController> {
    error("NavController не предоставлен")
}

@Composable
fun MyApp() {
    val navController = rememberNavController()
    CompositionLocalProvider(LocalNavController provides navController) {
        AppNavHost()
    }
}

// Использование глубоко в верстке:
@Composable
fun HotelCard(hotelId: String) {
    val navController = LocalNavController.current
    
    Card(onClick = { navController.navigate("details/$hotelId") }) {
        Text("Посмотреть отель")
    }
}
```
*Примечание: Сторонники абсолютно «чистой» архитектуры предпочитают прокидывать лямбды (коллбэки), чтобы UI компоненты вообще ничего не знали о навигации, но на практике `LocalNavController` экономит сотни часов разработки.*

### 3. Глобальный загрузчик изображений (Например, Coil)

Если ваше приложение требует авторизации для скачивания картинок, 
вам нужно добавлять токен в заголовки каждого запроса. 
Создавать такой клиент при загрузке каждой картинки неэффективно. 
Вы создаете один настроенный `ImageLoader` (с токенами, настроенным кэшем и логгером) и прокидываете его вниз.

```kotlin
val LocalImageLoader = staticCompositionLocalOf<ImageLoader> {
    error("ImageLoader не найден")
}

// В корне:
val customImageLoader = ImageLoader.Builder(context)
    .crossfade(true)
    .okHttpClient { /* Ваш клиент с токенами */ }
    .build()

CompositionLocalProvider(LocalImageLoader provides customImageLoader) {
    // Весь UI
}

// В верстке используем его:
AsyncImage(
    model = "https://my-api.com/image.jpg",
    imageLoader = LocalImageLoader.current,
    contentDescription = null
)
```

### 4. Feature Flags (Флаги конфигурации)

Если в приложении есть A/B тесты или функции, которые включаются удаленно (Remote Config), 
не нужно прокидывать объект конфигурации в каждый экран.

```kotlin
data class AppFeatures(
    val isNewBookingFlowEnabled: Boolean,
    val isPremiumBannerVisible: Boolean
)

val LocalAppFeatures = staticCompositionLocalOf { AppFeatures(false, false) }

@Composable
fun BookingScreen() {
    val features = LocalAppFeatures.current
    
    if (features.isNewBookingFlowEnabled) {
        NewBookingComponent()
    } else {
        OldBookingComponent()
    }
}
```

---

### Главное правило: Когда НЕ стоит использовать CompositionLocal?

Не используйте его для **обычных бизнес-данных** (например, `LocalUser.current` или `LocalHotelList.current`). Для этого есть ViewModel и State.

`CompositionLocal` идеален только для **сквозных инфраструктурных зависимостей**, которые могут понадобиться *любому* элементу на экране, независимо от его роли и местоположения в иерархии.
