# Тема приложения

## Цветовая палитра

Логика Material Design 3 (M3) построена на **семантических ролях**.  

Вы больше не красите «кнопку в красный, а фон в белый», вы задаете роль: «это акцентный элемент (`primary`), а это поверхность (`surface`)».   

А библиотека Compose сама знает, какие цвета брать для своих компонентов.

Давайте разберем, как именно компоненты Jetpack Compose читают вашу цветовую схему (`colorScheme`).

---

При разборе дизайн-системы — будь то на лекции, при подготовке к соревнованиям по мобильной разработке или в реальном коммерческом проекте — важно уметь декомпозировать макет на строгие слоты Material 3. Это спасает от хаотичного использования «пипетки» и хардкода цветов.

## Как читать логику дизайнера и распределять акцентные и кастомные цвета.

### 1. Как выбираются базовые фоновые цвета (Логика)

В M3 есть четкая иерархия слоев. Снизу вверх:

* **`background` (Фон):** Самый нижний слой. Базовый холст приложения.
    * *Как выбрать:* Светлая тема — чисто белый (`#FFFFFF`) или светло-серый.  
  Темная тема — очень темно-серый (`#121212`), но **не** чисто черный (`#000000`), так как на черном слишком сильный контраст с белым текстом (смазывается при скролле).
    * **`onBackground`:** Цвет текста/иконок, которые лежат прямо на фоне.
* **`surface` (Поверхность):** Листы "бумаги", лежащие на фоне. Карточки, меню, диалоги.
    * *Как выбрать:* В светлой теме часто совпадает с `background` (белый на белом), а отделяется тенью или границей.  
  В темной теме `surface` обычно чуть светлее `background` (например, `#1E1E1E`), чтобы показать, что элемент находится "выше" (ближе к зрителю).
    * **`onSurface`:** Основной текст на поверхностях.
* **`surfaceVariant` (Альтернативная поверхность):** Используется для выделения блоков без использования ярких брендовых цветов или теней.
    * *Как выбрать:* Светло-серый (светлая тема) или средне-темный серый (темная тема).
    * **`onSurfaceVariant`:** Второстепенный текст, иконки.

### 2. Как выбираются акцентные цвета (Бренд и Действия)

Эта группа цветов отвечает за узнаваемость приложения и ведет пользователя по главному сценарию. 
У каждого из них есть свой on-цвет (для текста/иконок поверх) и Container (приглушенная версия для крупных элементов).

#### primary (Главный акцент): Лицо бренда и цвет самых важных действий.

Где искать в Figma: Главные кнопки (Call to Action — например, «Забронировать», «Оплатить», «Войти»), 
активная иконка в нижней панели навигации (BottomNavigationBar), главные Floating Action Buttons (FAB).

#### primaryContainer: Менее насыщенная версия главного цвета. 

Где искать: Крупные выделенные карточки, фоны для важных уведомлений. Текст на нем будет цвета onPrimaryContainer.

#### secondary (Второстепенный акцент): Поддерживающий цвет.

Он не должен отвлекать от primary, но должен выделять интерактивные элементы.

Где искать в Figma: Чекбоксы, переключатели (Switch), ползунки, фильтры (Chips), второстепенные кнопки (например, «Отмена» или кнопки с обводкой). 
Часто это менее яркий оттенок главного цвета или комплементарный цвет.

#### tertiary (Третичный акцент / Контраст): Цвет-балансир. 

Используется для создания сильного визуального контраста и привлечения внимания к уникальным элементам.

Где искать в Figma: Бейджи «Новинка», «Скидка», «Промо», графики, иконки в настройках. Это элементы, которые должны выделяться, но не являются главной кнопкой целевого действия.

### 3. Семантические цвета (Состояния)

   Эти цвета передают смысл независимо от бренда.

#### error (Ошибка): Сигнал о проблеме или деструктивном действии.

Где искать в Figma: Красный текст под полем ввода при неверном пароле, иконка корзины, кнопка «Удалить аккаунт».

### 4. Кастомные цвета (Когда не хватает Material 3)

   Стандартная палитра M3 покрывает 90% нужд, но в сложных интерфейсах всегда появляются цвета, 
   которые не вписываются в базовые слоты. 
   Пытаться «впихнуть» их в tertiary или surfaceTint — частая архитектурная ошибка.

Что это за цвета:

#### Статусы (Success, Warning, Info): Зеленый для успешной оплаты, желтый/оранжевый для ожидающей транзакции, синий для информационных плашек.

#### Бренды партнеров: Специфический красный для кнопки «Войти через Google» или синий для VK.

#### Градиенты или цвета уровней лояльности: Золотой, платиновый и т.д.

Где искать в Figma: Теги статусов заказов, иконки авторизации сторонних сервисов.

## Как профессионально искать цвета в Figma (Алгоритм)

Не используйте пипетку по экрану. Дизайнер мог наложить поверх карточки прозрачный слой или тень, и пипетка выдаст искаженный HEX-код.

**Ищите Local Styles (Локальные стили)**. 

Нажмите в пустую область макета. В правой панели Figma появится блок Local styles -> Color. 

Грамотный дизайнер всегда собирает палитру там. Вы увидите список вроде Primary / 500, Surface / Background, Semantic / Success.

Смотрите **панель Inspect (или Dev Mode)**. Выделив кнопку, смотрите в раздел Colors. 
Если цвет привязан к переменной (Token), вы увидите её название (например, md.sys.color.primary), 
что прямо указывает, в какой слот Compose его нужно положить.

### 5. Каким цветам соответствуют базовые компоненты Compose

Пусть мы используем Card

```kotlin
@Composable
fun Message(text: AnnotatedString, modifier: Modifier = Modifier) {
    Surface(
        modifier = modifier,
        color = Color.White,
        shape = RoundedCornerShape(6.dp),
        border = BorderStroke(1.dp, Color(0xFFE9E9E9))
    )
```

Эта функция задана так

```kotlin
@Composable
fun Surface(
  modifier: Modifier = Modifier,
  shape: Shape = RectangleShape,
  color: Color = MaterialTheme.colorScheme.surface, // <<<<<-- цвет фона
  contentColor: Color = contentColorFor(color), // <<<<<-- цвет содержимого
  // ... остальные параметры
)
```

Что позволяет нам систематизировать цвета и формы в теме. И упростить их использование.
Будет использован цвет по умолчанию, и форма - заданная в теме.

```kotlin
Surface(
  modifier = modifier,
  shape = MaterialTheme.shapes.small, //берем из нашей дизайн-системы M3
  // В философии Material Design 3 (M3) считается, что обводка контента (outline) 
  // почти всегда имеет фиксированную толщину 1.dp,
  // а меняется только её цвет (outline или outlineVariant).
  border = BorderStroke(1.dp, MaterialTheme.colorScheme.outlineVariant)
) 
```

Если вы не передаете компонентам кастомные цвета через `colors = ...Defaults.colors()`, они автоматически берут токены из вашей `MaterialTheme.colorScheme`.  

#### Вот шпаргалка по самым популярным компонентам.

####  Карточки (Cards)
В M3 есть три типа карточек, и они используют разные токены:

| Компонент Compose | Цвет фона | Цвет обводки | Цвет контента (текста) |
| :--- | :--- | :--- | :--- |
| **`ElevatedCard`** | `surface` | Нет | `onSurface` |
| **`FilledCard`** | `surfaceVariant` | Нет | `onSurface` |
| **`OutlinedCard`** | `surface` | `outline` | `onSurface` |

*(Примечание: В M3 `ElevatedCard` использует тональное возвышение (Tonal Elevation) — чем больше вы зададите параметр `elevation`, тем сильнее цвет `surface` будет смешиваться с цветом `primary`, приобретая легкий оттенок бренда).*

####  Текстовые поля (TextFields)

| Компонент Compose | Фон (когда не в фокусе) | Линия / Обводка | Текст | Ярлык (Label) / Подсказка |
| :--- | :--- | :--- | :--- | :--- |
| **`OutlinedTextField`** | Прозрачный | `outline` (Фокус: `primary`) | `onSurface` | `onSurfaceVariant` |
| **`TextField` (Filled)** | `surfaceVariant` | `onSurfaceVariant` (Фокус: `primary`) | `onSurface` | `onSurfaceVariant` |

####  Кнопки (Buttons)

| Компонент Compose | Цвет фона | Цвет текста / иконки |
| :--- | :--- | :--- |
| **`Button`** (Основная) | `primary` | `onPrimary` |
| **`FilledTonalButton`** | `secondaryContainer` | `onSecondaryContainer` |
| **`OutlinedButton`** | Прозрачный (Обводка: `outline`) | `primary` |
| **`TextButton`** | Прозрачный | `primary` |
| **`FloatingActionButton` (FAB)** | `primaryContainer` | `onPrimaryContainer` |

#### 📱 Структурные элементы

| Компонент Compose                                      | Цвет фона | Примечание |
|:-------------------------------------------------------| :--- | :--- |
| **`Scaffold`**                                         | `background` | Автоматически красит весь экран. |
| **`TopAppBar`**                                        | `surface` | При скролле контента под ним слегка меняет оттенок (Tonal Elevation). |
| **`BottomAppBar`** / `NavigationBar`                   | `surface` или `surfaceContainer`| Использует `primary` или `secondary` для активных вкладок. |
| **`Snackbar`** (Тост внизу)                            | `inverseSurface` | Текст берется из `inverseOnSurface` (темно-серый фон со светлым текстом в светлой теме). |
| **`HorizontalDivider, VerticalDivider`** (Разделитель) | `outlineVariant` | Тонкая, едва заметная линия. |

### 6. Важные системные цвета для границ и линий

Вы могли заметить токены `outline` и `outlineVariant`. Они очень важны для чистоты интерфейса:

* **`outline`:** Используется для четких границ элементов, с которыми пользователь взаимодействует (обводка `OutlinedTextField`, обводка `OutlinedButton`, границы `OutlinedCard`). Должен быть достаточно контрастным к фону.
* **`outlineVariant`:** Используется для декоративных линий, которые структурируют контент, но не требуют взаимодействия (разделители в списках `Divider`). Должен быть тусклым, чтобы не отвлекать внимание.

## Итог: правило "Поверхность + Контент"

Логика Compose всегда работает парами: если компонент решает использовать для своего фона `surfaceVariant`, он **автоматически** возьмет для текста и иконок внутри себя `onSurfaceVariant` (через механизм `LocalContentColor`). Вам не нужно красить текст внутри карточки вручную, если схема настроена правильно.

## 2. Border

В стандартном Material 3 нет встроенного объекта для хранения толщины и стилей обводок. 
Но хардкодить `BorderStroke(1.dp, Color.Gray)` прямо в верстке — это плохая практика, которая ломает поддержку темной темы и усложняет рефакторинг.

Чтобы систематизировать обводки профессионально, мы не просто создаем функции, которые перерисовываются при каждом вызове, а используем механизм `CompositionLocal`. 
Это позволяет один раз "собрать" стили при инициализации темы и эффективно отдавать их любому компоненту.

### Шаг 1: Создаем классы для хранения

```kotlin
//import androidx.compose.foundation.BorderStroke
//import androidx.compose.material3.MaterialTheme
//import androidx.compose.runtime.Composable
//import androidx.compose.runtime.CompositionLocalProvider
//import androidx.compose.runtime.ReadOnlyComposable
//import androidx.compose.runtime.remember
//import androidx.compose.runtime.staticCompositionLocalOf
//import androidx.compose.ui.graphics.Color
//import androidx.compose.ui.unit.dp

// 1. Описываем, какие обводки есть в нашем приложении
data class AppBorders(
  val thin: BorderStroke,
  val active: BorderStroke,
  val error: BorderStroke
)

// 2. Создаем Local-провайдер с пустыми значениями по умолчанию (заглушками)
val LocalAppBorders = staticCompositionLocalOf {
  AppBorders(
    thin = BorderStroke(0.dp, Color.Unspecified),
    active = BorderStroke(0.dp, Color.Unspecified),
    error = BorderStroke(0.dp, Color.Unspecified)
  )
}

// 3. Делаем красивое расширение для MaterialTheme, чтобы вызывать обводки привычным образом
val MaterialTheme.borders: AppBorders
@Composable
@ReadOnlyComposable
get() = LocalAppBorders.current
```

### Шаг 2: Внедряем в главную тему приложения

Так как цвета обводок зависят от того, какая тема сейчас включена (светлая или темная), мы собираем наши обводки внутри функции AppTheme, опираясь на актуальную палитру colorScheme.

```kotlin
@Composable
fun AppTheme(
    darkTheme: Boolean = false,
    content: @Composable () -> Unit
) {
    val colorScheme = if (darkTheme) DarkColors else LightColors
    
    // Пересоздаем обводки ТОЛЬКО если изменилась цветовая схема
    val borders = remember(colorScheme) {
        AppBorders(
            thin = BorderStroke(1.dp, colorScheme.outlineVariant),
            active = BorderStroke(2.dp, colorScheme.primary),
            error = BorderStroke(2.dp, colorScheme.error)
        )
    }

    // Передаем наши цвета, формы и кастомные обводки вниз по дереву
    MaterialTheme(
        colorScheme = colorScheme,
        shapes = AppShapes
    ) {
        CompositionLocalProvider(
            LocalAppBorders provides borders,
            content = content
        )
    }
}
```

### Шаг 3: Как это использовать:

Теперь обводка сама сменит цвет при переключении на темную тему, а движок Compose не потратит лишних ресурсов на ее пересчет.

```kotlin
@Composable
fun Message(text: AnnotatedString, modifier: Modifier = Modifier) {
  Surface(
    modifier = modifier,
    shape = MaterialTheme.shapes.small,
    // Вызываем нашу систематизированную обводку
    border = MaterialTheme.borders.thin
  ) {
    // ...
  }
}
```

## 3. Новый раздел: Spacing (Система отступов)

Spacing (Система отступов)
Одна из самых частых ошибок в верстке — использование произвольных отступов (padding(13.dp), padding(7.dp)). Это делает интерфейс неопрятным и разрушает визуальную ритмику.

В мобильной разработке золотым стандартом является 8-пиксельная сетка (8-pt grid system). 
Все отступы и размеры элементов должны быть кратны 8 (или 4 для мелких деталей). 
Это правило гарантирует, что дизайн будет выглядеть гармонично на экранах любой плотности.

Мы можем зашить эту сетку прямо в нашу дизайн-систему, используя тот же оптимизированный подход CompositionLocal.

### Шаг 1: Создаем структуру отступов
```kotlin
//import androidx.compose.material3.MaterialTheme
//import androidx.compose.runtime.Composable
//import androidx.compose.runtime.ReadOnlyComposable
//import androidx.compose.runtime.staticCompositionLocalOf
//import androidx.compose.ui.unit.Dp
//import androidx.compose.ui.unit.dp

// 1. Описываем нашу 8-пиксельную сетку
data class AppSpacing(
    val default: Dp = 0.dp,
    val extraSmall: Dp = 4.dp,
    val small: Dp = 8.dp,
    val medium: Dp = 16.dp,  // Базовый отступ от краев экрана
    val large: Dp = 24.dp,
    val extraLarge: Dp = 32.dp,
    val huge: Dp = 48.dp
)

// 2. Создаем Local-провайдер. 
// Так как отступы не меняются от светлой/темной темы, мы сразу передаем готовый объект.
val LocalAppSpacing = staticCompositionLocalOf { AppSpacing() }

// 3. Добавляем расширение к MaterialTheme
val MaterialTheme.spacing: AppSpacing
    @Composable
    @ReadOnlyComposable
    get() = LocalAppSpacing.current
```
### Шаг 2: Внедряем в AppTheme

Вам нужно добавить всего одну строчку в ваш CompositionLocalProvider внутри главной темы:

```kotlin
CompositionLocalProvider(
    LocalAppBorders provides borders,
    LocalAppSpacing provides AppSpacing(), // <-- Добавили провайдер отступов
    content = content
)
```

### Шаг 2: Используем

Вы больше не должны задумываться, какой отступ поставить — 15 или 16 dp. ВЫ оперируете семантическими терминами (маленький, средний, большой).

```kotlin
Column(
    // Внешний отступ экрана = 16.dp
    modifier = Modifier.padding(MaterialTheme.spacing.medium), 
    // Расстояние между элементами внутри колонки = 8.dp
    verticalArrangement = Arrangement.spacedBy(MaterialTheme.spacing.small) 
) {
    Text("Заголовок")
    
    Spacer(modifier = Modifier.height(MaterialTheme.spacing.large)) // Отступ 24.dp
    
    Button(onClick = { /*...*/ }) {
        Text("Подтвердить")
    }
}
```

Преимущество подхода: Если арт-директор решит, что в новом дизайне приложения базовый отступ экрана должен стать более "воздушным" (например, 20.dp вместо 16.dp), вы измените значение только в одном месте — в data class AppSpacing. Весь проект адаптируется автоматически.
