# Как систематизировать данные, клиентско-серверного приложения

> **"Выполни анализ файлов RoomData и LocalDataProvider.kt c точки зрения организации SDUI для этих данных, выскажи критику "**

Эти файлы представляют собой хорошую отправную точку для создания слоя данных (Data Layer) и мок-репозитория. 

Использование `Result<T>` и `suspend`-функций с `delay` для имитации сетевых задержек — это классический и правильный подход при **прототипировании**.

Однако, если мы оцениваем этот код с точки зрения масштабируемой коммерческой разработки и подготовки к **Server-Driven UI (SDUI)**, в архитектуре есть серьезные логические нарушения.

Ниже приведен подробный разбор, критика и план трансформации этих данных под SDUI.

---

### 1. Критика текущей архитектуры

#### Ошибка 1: Смешивание Data-слоя и UI-логики (Нарушение SOLID)
В файле `RoomData.kt` дата-классы содержат логику форматирования строк:
```kotlin
// В HotelData:
val formattedDistance: String
    get() = if (distanceFromCentre >= 1000) { ... } else { "$distanceFromCentre m from centre" }

// В OfferStateData:
val ratingText: String
    get() = when {
        score > 8 -> "Fabulous"
        score > 7 -> "Good"
        // ...
    }
```
**В чем проблема:** Дата-классы (DTO или Domain-модели) должны быть "глупыми". Они должны только хранить данные. Если завтра маркетологи решат, что рейтинг > 8 должен называться не "Fabulous", а "Excellent", вам придется выпускать обновление всего приложения. Кроме того, как вы сами отметили в комментариях, жесткая привязка к строкам делает невозможной нормальную локализацию (перевод на другие языки).

#### Ошибка 2: «Божественный» вторичный конструктор
В классе `TradeOfferData` написан гигантский вторичный конструктор на 15 параметров.
**В чем проблема:** Это антипаттерн "Telescoping Constructor". Он жестко связывает родительский класс со структурой всех вложенных классов (`RoomData`, `PriceDetailData`, `OfferStateData`). Если изменится структура `PriceDetailData` (например, добавится налог), вам придется переписывать этот огромный конструктор. Для сборки сложных объектов лучше использовать паттерн Фабрика (Factory) или мапперы.

#### Ошибка 3: Синглтон-провайдер
Использование `object LocalDataProvider : DataProviderI` лишает вас гибкости.
**В чем проблема:** В современных приложениях репозитории создаются и внедряются через Dependency Injection (Dagger/Hilt/Koin). Использование `object` делает код трудно тестируемым (невозможно подменить реализацию в Unit-тестах без костылей).

### 2. Как исправить текущий код (Классический подход)

Если вы пока не переходите на SDUI, а просто чистите архитектуру:

1. **Удалите форматирование из Data-классов.** 
2. **Перенесите логику в ViewModel или UI-мапперы.** ViewModel должна получать `distanceFromCentre = 1400`, прогонять через ресурсы Android (`R.string.distance_km`) и отдавать во View уже готовую строку `"1.4 км от центра"`.
3. **Используйте DI:** Замените `object LocalDataProvider` на `class LocalDataProvider(...)` и передавайте его как зависимость.

### 3. Переход к Server-Driven UI (SDUI)



Если ваша конечная цель — **SDUI**, текущие дата-классы придется полностью переосмыслить.

Суть SDUI в том, что мобильное приложение вообще не должно знать, что такое `distanceFromCentre`, как его делить на 1000 и когда писать "Fabulous". **Сервер присылает готовые компоненты и готовые строки.**

#### Шаг 1: Меняем JSON-ответ сервера
Вместо того чтобы сервер присылал "сырые" данные:
```json
// ПЛОХО (Текущий подход):
{
  "hotelName": "Lindala",
  "score": 8.6,
  "distanceFromCentre": 450
}
```
Сервер должен присылать UI-компоненты:
```json
// ОТЛИЧНО (SDUI подход):
{
  "type": "HOTEL_CARD",
  "id": "12345",
  "content": {
    "title": "Lindala",
    "badge_primary": {
      "text": "8.6",
      "label": "Fabulous"
    },
    "subtitle_icon": "ic_location",
    "subtitle_text": "450 m from centre",
    "price_strikethrough": "€450",
    "price_main": "€390"
  }
}
```

#### Шаг 2: Создаем классы-компоненты в Kotlin
Ваши дата-классы больше не описывают бизнес-логику (отели, налоги, расстояния). Они описывают **UI-элементы**.

```kotlin
// Базовый класс для всех компонентов, приходящих с сервера
sealed class ServerUIComponent {
    abstract val id: String
}

// Модель для карточки отеля (Она "глупая", только строки!)
data class HotelCardComponent(
    override val id: String,
    val title: String,
    val badgeText: String,     // Сервер сам пришлет "8.6"
    val badgeLabel: String?,   // Сервер сам пришлет "Fabulous" или null
    val subtitleText: String,  // Сервер сам пришлет "450 m from centre"
    val mainPrice: String,
    val oldPrice: String?,
    val promoTag: PromoTagComponent? // Вложенный SDUI компонент
) : ServerUIComponent()

data class PromoTagComponent(
    val text: String,
    val isAccented: Boolean
)
```

#### Шаг 3: Маппинг и Отрисовка в Compose
В Compose вы пишете фабрику, которая читает тип компонента и рендерит нужную карточку.

```kotlin
@Composable
fun SDUIHost(components: List<ServerUIComponent>) {
    LazyColumn {
        items(components) { component ->
            when (component) {
                is HotelCardComponent -> {
                    // Передаем готовые строки прямо в UI! Никакой математики.
                    HotelListingCard(
                        title = component.title,
                        price = component.mainPrice,
                        ratingLabel = component.badgeLabel,
                        // ...
                    )
                }
                // is BannerComponent -> BannerTemplate(...)
                // is DividerComponent -> HorizontalDivider()
            }
        }
    }
}
```

### Итог

Ваш текущий файл `RoomData.kt` содержит классические *Domain Models* (доменные модели). Это нормально для стандартного приложения.

Но для **SDUI** вам нужно вынести всю логику вычислений (`score > 8`, `distance / 1000`) на backend. Мобильное приложение должно получать от API `LocalDataProvider` уже финальные, отформатированные локализованные строки, разложенные по полочкам UI-компонентов. Это позволит вам мгновенно менять дизайн, тексты и условия акций, не перекомпилируя приложение.
